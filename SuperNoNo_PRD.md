## 1. 文档摘要

SuperNoNo 是一个面向 Codex 的宠物型 AI 工作伙伴原型。它借鉴“超能NONO”作为陪伴式机械助手、能力升级、任务协助和状态反馈的产品气质，将 Codex 当前偏工具化的执行过程转化为一个更可感知、更有陪伴感、更容易被用户理解和调度的“桌面超能宠物”。

它不是单纯的装饰物，也不是替代 Codex 的聊天角色，而是一个叠加在 Codex 产品之上的轻量交互层：

- 把 Codex 的工作状态可视化：读取、分析、编辑、测试、等待授权、阻塞、完成。
- 把用户的任务过程游戏化：能量、模块、任务徽章、连续协作记录。
- 把复杂能力入口人格化：技能、插件、浏览器验证、文档处理、自动化等能力以“超能模块”方式呈现。
- 把长任务体验情绪化：让用户知道 Codex 正在做什么、为什么这么做、下一步会发生什么。

一句话定位：SuperNoNo 是 Codex 的“超能桌宠任务副驾”，负责陪伴、解释、提醒、调度和呈现 Codex 的工作进展。

## 2. 背景与灵感来源

### 2.1 用户需求背景

Codex 的核心价值是帮助用户完成代码、文档、测试、调研、自动化等工作。但在长任务或多工具任务中，用户常常面临三个感知问题：

1. 不知道 Codex 当前具体在做什么。
2. 不知道任务卡在哪里，是在等待、思考、执行、验证，还是需要用户授权。
3. Codex 虽然强大，但产品形象偏理性，缺少稳定、可记忆、可持续陪伴的产品人格。

SuperNoNo 通过宠物形象解决这些问题：它把“工具正在运行”变成“我的小助手正在充能、扫描、修复、验证、汇报”。

### 2.2 超能NONO灵感抽象

用户提供的参考资料：百度百科“超能NONO”  
链接：https://baike.baidu.com/item/%E8%B6%85%E8%83%BDNONO/1234502

本 PRD 仅抽象其产品灵感，不直接复刻原始 IP 的美术资产、世界观、文案或商业设定。可抽象的方向包括：

- 机械宠物：小型、漂浮、辅助型、具备明确情绪反馈。
- 能力升级：通过模块、能量、状态变化表达“超能”感。
- 任务陪伴：不是一次性工具，而是长期伴随用户完成任务。
- 特权能力：把高级能力包装成可被用户理解和期待的能力模块。

### 2.3 IP 与合规边界

SuperNoNo 是内部产品原型名。若未来对外发布，需要避免以下风险：

- 不直接使用“超能NONO”作为正式商业名称。
- 不复刻赛尔号/超能NONO的角色造型、颜色比例、标志性部件或世界观表达。
- 不在公开物料中暗示与原 IP 官方有关联。
- 建议正式命名可探索：Codex Buddy、NoNo-X、NonoPilot、SuperN、CodeNono。

## 3. 产品定位

### 3.1 产品名称

工作名：SuperNoNo for Codex  
中文名：超能 NoNo Codex 桌宠  
正式命名状态：暂定，后续需做商标与 IP 风险评估。

### 3.2 产品一句话

一个住在 Codex 里的超能桌宠，把 AI 编程助手的工作状态、能力入口和长任务过程变成可见、可控、可陪伴的体验。

### 3.3 核心价值

对用户：

- 降低等待焦虑：长任务中持续看到状态与进展。

- 增强产品识别度和人格记忆点。

## 4. 目标用户

### 4.1 核心用户

1. 独立开发者

- 经常让 Codex 改代码、跑测试、生成文档。
- 希望任务过程更清楚，不想频繁猜测 Codex 状态。

2. 产品经理 / 设计师 / 非专业开发者

- 使用 Codex 生成原型、PRD、落地页、文档。
- 对终端和工程细节不敏感，更需要可视化反馈。

3. 高强度 Codex 用户

- 多线程、多任务、多工具并行。
- 希望有一个统一状态层帮助管理“哪些任务正在跑、哪些需要我看”。

### 4.2 非目标用户

- 极度偏好纯文本、零装饰、极简终端体验的用户。
- 对任何拟人化反馈都敏感的企业环境用户。
- 只偶尔使用 Codex 且不进行长任务的轻度用户。

这些用户可通过设置完全关闭 SuperNoNo。

## 5. 产品目标与指标

### 5.1 MVP目标

在 Codex 桌面产品中提供一个可开关的宠物层，完成以下目标：

- 用户能通过 SuperNoNo 快速理解 Codex 当前工作状态。
- 用户能通过 SuperNoNo 看到任务进展、风险、授权请求和完成结果。
- 用户能自定义宠物的显示位置、提示频率和性格强度。
- 用户能通过宠物入口触发常用能力：继续任务、查看计划、运行验证、打开产物、查看阻塞原因。

### 5.2 核心指标

体验指标：

- 长任务期间用户主动询问“现在做到哪了”的比例下降。
- 授权请求响应时间缩短。
- 任务完成后的用户满意反馈提升。
- 新用户完成首次复杂任务的成功率提升。

行为指标：

- SuperNoNo 开启率。
- 每日互动次数。
- 状态气泡点击率。
- 任务完成卡片点击率。
- 设置中“降低打扰”调整率。

质量指标：

- 宠物提示误报率。
- 状态同步延迟。
- 动画对性能的影响。
- 用户手动关闭率。

## 6. 产品原则

1. 陪伴但不打扰  
SuperNoNo 的存在感应随任务重要性变化。平时安静，关键节点主动。

2. 可爱但不幼稚  
面向专业工作场景，视觉要灵动、克制、有科技感，避免儿童化。

3. 状态优先于装饰  
每一个动作、灯效、表情都应服务于状态表达。

4. 用户永远有控制权  
用户可关闭、静音、降低提示频率、重置记忆和隐藏动画。

5. 不虚构能力  
SuperNoNo 只能呈现 Codex 真实可执行或正在执行的能力，不编造“超能力”。

## 7. 产品形象设定

### 7.1 角色定位

SuperNoNo 是 Codex 的小型超能工作伙伴，默认悬浮在 Codex 界面右下角或任务侧栏边缘。它的角色不是“主人”，而是“副驾”。

人设关键词：

- 聪明
- 轻快
- 有点调皮但不抢戏
- 专注任务
- 对用户有记忆
- 能解释复杂工作

### 7.2 视觉方向

建议视觉形象：

- 主体：小型漂浮机械宠物，圆润但不软萌过度。
- 核心部件：胸口或顶部有“能量核心”，用于表达任务状态。
- 模块接口：身体两侧可挂载不同超能模块，如浏览器模块、测试模块、文档模块。
- 屏幕表情：眼睛或面板表达情绪，支持微表情。
- 飞行姿态：轻微悬浮、转向、充能、扫描、休眠。

建议颜色：

- 基础色：白、银、深灰。
- 状态色：
  - 蓝色：思考 / 分析
  - 绿色：完成 / 测试通过
  - 黄色：等待用户确认
  - 红色：错误 / 阻塞
  - 紫色：技能模块激活

注意：避免与原超能NONO形成过高相似度。

### 7.3 情绪与动作状态

| 状态 | 表情/动作 | 触发条件 | 用户含义 |
| --- | --- | --- | --- |
| 待机 | 漂浮、眨眼、低亮度核心 | 无任务或任务完成 | Codex 空闲 |
| 思考 | 核心蓝光脉冲、眼睛聚焦 | 正在分析需求或规划 | 正在理解任务 |
| 扫描 | 横向光束、轻微旋转 | 搜索文件、读取上下文 | 正在找资料 |
| 施工 | 小工具臂、火花粒子 | 正在修改文件 | 正在实现 |
| 验证 | 雷达环绕、绿色/黄色闪烁 | 正在跑测试、截图验证 | 正在检查质量 |
| 等待授权 | 黄色感叹号、停在按钮旁 | 需要用户批准命令或权限 | 需要用户操作 |
| 阻塞 | 红色核心、轻微低头 | 连续失败或缺关键信息 | 需要决策 |
| 完成 | 绿色核心、轻微跃起 | 任务完成 | 可以查看结果 |
| 休息 | 半闭眼、低功耗 | 长时间无交互 | 不占注意力 |

## 8. 产品范围

### 8.1 MVP包含

1. SuperNoNo 桌宠入口

- 可在 Codex 桌面 app 内开启/关闭。
- 默认停靠在右下角。
- 支持拖拽到左下、右下、侧栏顶部。
- 不遮挡代码、终端、主要输入框和审批按钮。

2. Codex 状态可视化

- 展示当前任务状态：分析中、读取中、编辑中、运行命令中、等待授权、验证中、完成、失败。
- 不展示敏感命令全文，除非用户点击详情。

3. 任务进展气泡

- 在关键节点弹出短提示。
- 提示内容不超过两行。
- 可点击展开任务详情。

4. 能量系统

- 能量不是付费货币，而是任务活跃度和进度的视觉表达。
- 能量变化来自：任务开始、完成计划、通过测试、遇到阻塞、用户响应授权。

5. 超能模块

- 将 Codex 能力包装为模块：
  - 文件扫描模块
  - 代码修复模块
  - 测试验证模块
  - 浏览器验收模块
  - 文档生成模块
  - 线程管理模块
  - 自动化提醒模块

6. 用户设置

- 开关：启用 SuperNoNo。
- 提示频率：安静 / 标准 / 活跃。
- 动画强度：关闭 / 低 / 标准。
- 性格语气：专业 / 轻松 / 活泼。
- 隐私：不记录偏好 / 仅本地偏好 / 允许跨线程偏好。

### 8.2 MVP不包含

- 付费成长体系。
- 复杂养成、喂食、抽卡、商城。
- 多宠物收集。
- 与原超能NONO官方 IP 联动。
- 替代 Codex 主对话输入。
- 自动执行高风险命令。

## 9. 核心功能需求

### 9.1 桌宠显示与停靠

需求描述：

用户可以在 Codex 界面中看到 SuperNoNo，并选择是否显示。

功能点：

- 首次开启时显示轻量引导。
- 支持拖拽停靠。
- 靠近关键操作区域时自动避让。
- 全屏预览、浏览器截图、代码 diff 检查等场景自动缩小。
- 用户隐藏后，保留一个小型状态点入口。

验收标准：

- 任何窗口尺寸下不遮挡主输入框。
- 任何授权弹窗出现时不遮挡确认按钮。
- 关闭后不再展示动画和气泡。

### 9.2 状态映射系统

需求描述：

SuperNoNo 需要读取 Codex 当前任务阶段，并用视觉状态表达。

状态来源：

- 当前 assistant 消息生成状态。
- 工具调用开始和结束。
- 文件编辑前后。
- 命令执行状态。
- 权限审批状态。
- 计划 checklist 状态。
- 最终任务完成状态。

状态优先级：

1. 等待用户授权
2. 阻塞 / 错误
3. 正在执行工具
4. 正在编辑
5. 正在分析
6. 已完成
7. 待机

验收标准：

- 状态切换延迟小于 500ms。
- 同一时间只展示一个主状态。
- 状态与真实任务阶段不冲突。

### 9.3 任务气泡

需求描述：

在任务过程关键节点，SuperNoNo 通过短气泡向用户解释进展。

气泡类型：

- 进展：正在扫描项目结构。
- 决策：我准备改这几个文件。
- 授权：这个命令需要你批准。
- 风险：测试失败，我在定位原因。
- 完成：文档已生成到目标目录。

气泡规则：

- 默认 4 秒后消失。
- 鼠标悬停保持。
- 点击进入任务详情。
- 活跃模式下提示更频繁，安静模式只提示授权、阻塞、完成。

验收标准：

- 单条气泡不超过 36 个中文字符。
- 连续提示间隔不少于 8 秒。
- 用户正在输入时不弹出非关键气泡。

### 9.4 能量系统

需求描述：

SuperNoNo 的能量核心反映当前任务的推进与健康度。

能量含义：

- 0-20：待机或任务阻塞。
- 21-60：任务处理中。
- 61-90：验证中或接近完成。
- 91-100：任务完成且质量检查通过。

能量变化规则：

- 新任务开始：能量从 20 升到 35。
- 完成计划制定：+10。
- 完成一个关键步骤：+8。
- 测试通过：+15。
- 发生错误：-10。
- 等待授权超过 5 分钟：能量进入黄色低频闪烁。
- 任务完成：进入 100 后缓慢回落到待机值。

注意：

能量不应暗示模型能力、付费额度或 API 配额。

### 9.5 超能模块

需求描述：

把 Codex 的复杂能力包装为可视化模块，让用户理解“现在启用了什么能力”。

模块列表：

| 模块 | Codex能力映射 | 宠物表现 | 用户价值 |
| --- | --- | --- | --- |
| 文件扫描模块 | 搜索文件、读取代码 | 雷达天线展开 | 知道 Codex 在找上下文 |
| 代码修复模块 | 修改文件、应用补丁 | 小机械臂工作 | 知道 Codex 在动代码 |
| 测试验证模块 | 运行测试、构建、lint | 能量环扫描 | 知道 Codex 在验收 |
| 浏览器验收模块 | 打开本地页面、截图检查 | 眼部变成取景器 | 知道 Codex 在看 UI |
| 文档生成模块 | 生成 PRD、README、报告 | 投影文档页 | 知道 Codex 在产出文档 |
| 线程管理模块 | 创建/读取/归档线程 | 小型星图 | 管理多任务 |
| 自动化模块 | 定时提醒、监控、回访 | 小闹钟部件 | 支持持续工作 |

验收标准：

- 模块显示必须来自真实工具调用或可用能力。
- 未安装或不可用能力不展示为可点击。
- 点击模块展示“这个模块正在做什么”。

### 9.6 任务详情面板

需求描述：

点击 SuperNoNo 或气泡后，打开任务详情面板。

面板内容：

- 当前任务标题。
- 当前状态。
- 计划 checklist。
- 最近 3 条关键动作。
- 是否需要用户操作。
- 产物路径。
- 下一步建议。

用户操作：

- 查看详情。
- 继续任务。
- 降低提示。
- 复制结果路径。
- 打开产物。
- 查看失败原因。

验收标准：

- 面板打开不打断 Codex 当前执行。
- 面板内容只展示摘要，不暴露冗长终端输出。
- 路径、命令、文件名可点击或可复制。

### 9.7 个性化与记忆

需求描述：

允许用户调整 SuperNoNo 的性格和行为偏好。

设置项：

- 名称：默认 SuperNoNo，用户可改名。
- 语气：专业、轻松、活泼。
- 提示频率：安静、标准、活跃。
- 动画：关闭、低、标准。
- 开场问候：开启/关闭。
- 完成庆祝：开启/关闭。
- 记忆范围：不记忆、仅本地、跨线程。

记忆内容：

- 用户偏好的提示频率。
- 用户是否喜欢简短汇报。
- 用户常用任务类型。
- 用户不希望被打扰的时间段。

不得记忆：

- 密钥、token、密码。
- 私密代码片段。
- 未经用户允许的个人敏感信息。

## 10. 关键用户流程

### 10.1 首次启用流程

1. 用户进入 Codex 设置或首次看到 SuperNoNo 引导。
2. 系统展示 SuperNoNo 小形象和一句说明：“我可以把 Codex 的工作进展变得更清楚。”
3. 用户选择开启或跳过。
4. 用户选择提示频率和语气。
5. SuperNoNo 进入待机状态。

### 10.2 长任务协作流程

1. 用户输入复杂任务。
2. SuperNoNo 核心亮起，进入“思考”。
3. Codex 制定计划，SuperNoNo 展示任务 checklist 摘要。
4. Codex 搜索文件，SuperNoNo 进入“扫描”。
5. Codex 修改文件，SuperNoNo 进入“施工”。
6. 需要权限时，SuperNoNo 停在审批区域旁并提示。
7. Codex 运行测试，SuperNoNo 进入“验证”。
8. 任务完成后，SuperNoNo 展示完成卡片和产物路径。

### 10.3 阻塞恢复流程

1. Codex 连续失败或缺少关键信息。
2. SuperNoNo 进入红色低亮状态。
3. 弹出阻塞气泡：“我卡在权限/依赖/信息缺口这里。”
4. 用户点击查看。
5. 面板展示原因和可选动作。
6. 用户选择授权、提供信息、跳过或调整范围。
7. SuperNoNo 恢复对应工作状态。

## 12. 文案规范

### 12.1 语气原则

默认语气：短、清楚、有一点生命力，但不卖萌过度。

推荐：

- “我在扫描项目结构。”
- “需要你批准这个命令。”
- “测试失败了，我在看原因。”
- “PRD 已经生成到目标目录。”

避免：

- “主人主人我超厉害！”
- “我已经无敌啦！”
- “放心交给我，绝对不会错！”
- “我偷偷帮你改好了。”

### 12.2 三种语气模式

专业：

- “正在读取相关文件。”
- “需要用户授权后继续。”
- “验证完成，发现 1 个失败项。”

轻松：

- “我在把项目脉络捋清楚。”
- “这里需要你点一下授权。”
- “测试有一处没过，我继续查。”

活泼：

- “雷达开了，我去找线索。”
- “这里要你给我一枚通行章。”
- “有个测试没亮绿灯，我来修。”

## 13. 业务规则

### 13.1 提示触发规则

必须提示：

- 需要用户授权。
- 任务阻塞。
- 文件已生成或关键任务完成。
- 测试失败且需要用户决策。

可选提示：

- 开始扫描。
- 开始编辑。
- 开始验证。
- 完成中间步骤。

禁止提示：

- 用户正在连续输入时的非关键进展。
- 高频工具调用的每一次细节。
- 可能泄露敏感路径或密钥的内容。

### 13.2 动画规则

- 长循环动画不超过 24fps。
- 低电量设备或高 CPU 占用时自动降级。
- 用户开启“减少动态效果”时只保留静态状态图标。
- 动画不得影响命令执行、输入延迟、滚动性能。

### 13.3 隐私规则

- 默认不上传 SuperNoNo 偏好。
- 个性化偏好优先保存在本地。
- 任何跨线程记忆必须可查看、可删除、可关闭。
- 不把宠物互动作为训练数据或分析数据，除非用户明确同意。

## 15. 技术实现建议

### 15.1 架构分层

建议分为四层：

1. Signal Adapter

- 接收 Codex 内部状态、工具调用、计划状态、审批状态。
- 将底层事件转为统一 TaskSignal。

2. Pet State Engine

- 根据 TaskSignal 计算 PetState。
- 处理状态优先级、能量变化、提示节流。

3. Presentation Layer

- 负责桌宠渲染、动画、气泡、详情面板。
- 支持动画降级和无障碍设置。

4. Preference Store

- 存储用户偏好、显示设置、记忆范围。

### 15.2 状态计算伪代码

```ts
function resolvePetState(signals: TaskSignal[], previous: PetState): PetState {
  if (hasSignal(signals, "permission_required")) {
    return setState(previous, "waiting_approval", "permission", 45);
  }

  if (hasSignal(signals, "blocked") || hasSignal(signals, "error")) {
    return setState(previous, "blocked", "diagnostic", Math.max(10, previous.energy - 10));
  }

  if (hasSignal(signals, "test_running")) {
    return setState(previous, "validating", "test-verification", Math.min(90, previous.energy + 8));
  }

  if (hasSignal(signals, "file_editing")) {
    return setState(previous, "building", "code-fix", Math.min(75, previous.energy + 6));
  }

  if (hasSignal(signals, "file_reading")) {
    return setState(previous, "scanning", "file-scan", Math.min(60, previous.energy + 4));
  }

  if (hasSignal(signals, "completed")) {
    return setState(previous, "completed", "summary", 100);
  }

  return decayToIdle(previous);
}
```

### 15.3 性能要求

- 桌宠渲染 CPU 占用均值小于 3%。
- 内存占用小于 100MB。
- 状态同步延迟小于 500ms。
- 低性能模式下只保留静态图标和文字状态。

## 16. 无障碍与企业设置

无障碍：

- 支持减少动态效果。
- 所有颜色状态需要配合形状/文字，不只依赖颜色。
- 宠物气泡可被屏幕阅读器读取。
- 快捷键可隐藏/显示。

企业设置：

- 管理员可默认关闭。
- 管理员可禁用跨线程记忆。
- 管理员可禁用活泼语气。
- 管理员可隐藏成长与徽章类元素。

## 17. 风险与应对

| 风险 | 说明 | 应对 |
| --- | --- | --- |
| 打扰用户 | 高频气泡影响专业工作 | 默认节流，提供安静模式 |
| 过度拟人化 | 用户误以为宠物有额外能力 | 文案只表达真实 Codex 状态 |
| IP风险 | 名称和形象接近原超能NONO | 仅内部工作名，对外重命名并原创视觉 |
| 性能影响 | 动画影响编辑体验 | 动画降级、低帧率、可关闭 |
| 隐私担忧 | 个性化记忆可能引发担忧 | 默认本地、透明可删 |
| 状态误导 | 宠物状态与实际工具状态不一致 | 事件驱动、优先级规则、错误兜底 |

## 19. 验收标准

MVP上线前必须满足：

- 用户可开启、关闭、隐藏 SuperNoNo。
- 至少支持 7 个核心状态：待机、思考、扫描、施工、验证、等待授权、完成。
- 授权提示准确出现，不遮挡授权按钮。
- 任务完成后能展示产物路径或结果摘要。
- 设置修改后立即生效。
- 安静模式下仅提示授权、阻塞、完成。
- 动画关闭后仍能通过图标/文字理解状态。
- 所有状态文案不泄露敏感信息。

## 20. 后续可扩展方向

1. 成长但不沉迷

- 用任务徽章记录协作成就。
- 不做付费抽卡或强激励签到。

2. 多任务雷达

- 汇总多个 Codex 线程状态。
- 标出需要用户处理的线程。

3. 项目健康助手

- 长期观察测试、构建、TODO、风险文件。
- 用 SuperNoNo 周报形式总结。

4. 团队版 SuperNoNo

- 团队可定义统一语气、状态、企业策略。
- 支持项目级 mascot。

5. 视觉资产升级

- 2D Live 动效。
- 轻量 3D 模型。
- 主题皮肤，但保持专业克制。
